Bug report: game crashes on launch on high-RAM phones (Pixel 9 Pro XL)

**Device:** Pixel 9 Pro XL, 16GB RAM
**Game version:** 1.822 (build cea6b550-3db9-4510-864b-a1c2a0089fa7)

**What happens:**
Splash screen shows, gets into the loader scene, then just dies a few seconds later ŌĆö no error dialog, no ANR, just closes back to the home screen. Intermittent ŌĆö sometimes launches fine, but crashes reliably most of the time on this device (see the ASLR finding below for why).

**What I found in logcat:**
Right before it dies:
`Using memoryadresses from more that 16GB of memory`
*(typo is in the actual log line, not mine)*

then immediately:
```
ActivityManager: Process com.vnlentertainment.socclassic (pid 26405) has died: fg TOP
Zygote: Process 26405 exited due to signal 11 (Segmentation fault)
```

So it's a native segfault, not a Java exception ŌĆö nothing in `AndroidRuntime`/`FATAL EXCEPTION`, just straight SIGSEGV. Timing lines up with `/proc/meminfo` reporting `MemTotal` ~15.9GB on this device.

**Root cause, confirmed (not a guess):**
Pulled the APK and checked `libunity.so`. This build is on **Unity 2017.4.34f1** ŌĆö from 2018, years before Unity's engine handled large address spaces properly.

Found the exact crash string sitting right next to Unity's own memory allocator's error strings:
```
Could not allocate memory: System out of memory!
Trying to allocate: %zuB with %zu alignment. MemoryLabel: %s
Allocation happend at: Line:%d in %s
[ %s ] used: %zuB | peak: %zuB | reserved: %zuB
Using memoryadresses from more that 16GB of memory
```
So it's not IL2CPP's GC ŌĆö it's Unity's own native memory manager in `libunity.so`. That 2017-era allocator packs its allocation bookkeeping assuming addresses stay under ~16GB (2┬│Ōü┤). On a 16GB-RAM phone, the OS can hand back heap addresses above that range once the game loads its first asset bundle (right when `LoaderScene`/`bundleVer` load happens ŌĆö exactly when it crashes). When that happens the packed pointer representation overflows, and it segfaults almost immediately after logging that warning.

**Proven with a live test, not just theory:**
The crash is **ASLR-dependent** ŌĆö that's why it looked intermittent. Android randomizes heap placement on every launch; when it lands above that ~16GB ceiling, it segfaults, otherwise it doesn't.

To confirm: I disabled ASLR (`personality(ADDR_NO_RANDOMIZE)`) for just this app's process on my rooted device and ran it **10 times in a row ŌĆö 10/10 launches succeeded, zero crashes**, versus a reliable crash streak beforehand with ASLR on. This isn't a maybe anymore ŌĆö address randomization is the confirmed trigger.

**The actual fix:** needs a Unity version bump ŌĆö later versions rewrote this allocator to handle 64-bit/large address spaces properly, which is exactly why this bug class exists on 2017.4 LTS. I can't ship you an "ASLR off" patch since that's an OS-level flag outside your APK, not something togglable from your build ŌĆö but it tells you exactly where to look: whatever in your allocator/init code does 16GB-relative pointer math needs to either use the full 64-bit address instead of a truncated/packed one, or detect addresses above its assumed ceiling and retry the allocation instead of silently corrupting a pointer. This is going to keep hitting every current 16GB+ RAM flagship (Pixel 8/9 Pro, Pro XL, several Samsung/OnePlus phones too) and will only get worse as more phones ship with 16GB.
